Design Principles / Ideas

Outcomes wanted

Improve 3 things

  1. Improve data submitted
  2. Improve ability to update the register automatically
  3. Improve experience for the caseworker (staff)

Design Principles / User needs/desires

  • Minimal interaction with existing applications and processes - only where we absolutely have to
  • Everything needed to process the application is brought to the user
  • Users are legal technicians
  • Tools focused on the goal - processing mortgage appication (not a swiss army knife system - yet! or maybe never!)
  • Simplify language. clarity and accuracy of terminololgy
  • The register is the atomic unit. everyone (internal and external) gets the same thing. no different views for different people. If the way it is organised/shown doesn't work - change the atomic thing - for everyone.
  • Less is best - strip away everything not needed. start with too little and add.
  • More, interesting work / less admin
  • Confidence / trust in the systems. IT and in what we are asking them to do (process/practice)
  • Supported and listened to. Closer (direct) working relationship with developers.

UX Principles

  • What experienced designers thought
  • What we learnt from our users and stakeholders
  • What we learnt from cross-government

Everything i need - in one place

  • The system gives me what i need to do my job.
  • It brings what i need most to the fore, but lets me discover the rest when i need to.

The system supports me

  • I make legal/skilled decisions. The system automates the admin work.
  • It lets me do things my own way, but always generates consistent results.

I trust the system

  • I care about the integrity of the register.
  • I can rely on the system, and it supports me in our goal.
  • It is easy to use and reliable.

ideas 'framing' the casework processing part

  • We are getting mortgage applications from a few, known, lenders
  • Number of applications will be small. 1 to 100 per day
  • Internal processing team will be small, co-located, and co-located with the service development team - operate, iterate and support the application/service as one team.
  • We will be iterating/bug fixing/enhancing on a daily basis
  • If the new system/process cannot cope with an application we can run it through the existing systems/processes
  • First release could look / operate nothing like later releases (metamorphosis).
  • An aim (not the only) of first release is to learn what we can take away - what isn't technical work/legal judgement - to be able to remove that and leave technical skilled work for staff.
  • The system (and people) will learn during the running of the beta (see below)
